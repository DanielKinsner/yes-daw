# 0011. Asset→Clip→Project non-destructive indirection; stable 128-bit ULID IDs

- **Status:** Accepted
- **Date:** 2026-06-23
- **Deciders:** Dan (owner), build agent (H1)
- **Related:** ADR-0002 (clips reference assets; never edit audio in place), ADR-0010 (clip positions in
  ticks; `time_base`), ADR-0012 (how this persists),
  [build plan](../plans/2026-06-23-feat-yes-daw-architecture-roadmap-plan.md) decisions #7/#8,
  [deepening notes](../plans/2026-06-23-yes-daw-deepening-notes.md) → *Data model* and Conflict #5,
  `CONTEXT.md` (Asset, Clip, Project).

## Context

A hard rule (ADR-0002): clips reference assets non-destructively; we never edit the underlying audio in
place. The indirection layers — Asset (the audio), Clip (a placement that points into an Asset), Project
(the arrangement) — and the **identity** of every entity are written into schema v1 and into every saved
Project. Identity in particular is hard to reverse: once files exist with a given ID scheme, changing it
breaks templates, cross-project paste, and interchange. The ID mechanism was the one unpinned part
(deepening-notes Conflict #5) and is locked now.

## Options considered

**Indirection:** flat destructive edits vs Asset→Clip→Project indirection — the latter is mandated by
ADR-0002 and is not re-litigated here; this ADR records its exact shape.

**Identity:**
1. **SQLite `INTEGER PRIMARY KEY` (rowid).**
   - Pros: free, fast, simplest.
   - Cons: reuses rowid space after deletes (alias/collision under undo/redo re-insert); unique only
     within one DB — no cross-project identity for templates/paste/DAWproject. Rejected.
2. **Monotonic-never-reused `INTEGER` (persisted `next_id`).**
   - Pros: no reuse; trivial.
   - Cons: still only unique within one Project. Rejected once cross-project identity is in scope.
3. **128-bit ULID.**
   - Pros: never reused, unique across Projects, time-sortable; enables templates, cross-project paste,
     and clean interchange. Resolved 2026-06-23.
   - Cons: 16 bytes vs 8; a generator to own. Accepted.

## Decision

**Asset → Clip → Project indirection, with every entity keyed by a 128-bit ULID.**

- **Asset:** immutable, **content-hashed** (SHA-256 over source bytes). Copied into the bundle by default;
  one Asset backs many Clips. Never edited in place.
- **Clip:** a non-destructive placement that **references** an Asset — its own start/end (in ticks),
  gain, fades, and `time_base` (ADR-0010). Invariant: `clip.src_offset + clip.src_len ≤ asset.frames`
  (asserted; split's canonical identity is `right.srcOffset == left.srcOffset + left.srcLength` exactly).
- **Project:** owns the arrangement (tracks, clips, routing, automation, node state); it is the unit that
  saves and round-trips.
- **Identity = 128-bit ULID,** never reused, unique across Projects. Generated by one allocator. Deletes
  of an Asset that backs Clips go through the undo/command layer (capture rows first), **not** raw FK
  cascades, during interactive editing.

## Consequences

- **Positive:** editing is non-destructive by construction; content-hash dedupe means one physical copy
  per unique Asset; stable cross-project IDs unlock templates, paste-between-projects, and lossless
  interchange; split is exact and reversible.
- **Negative / accepted costs:** 16-byte IDs; a ULID generator to own and test; the `src_offset+len ≤
  frames` and split-exactness invariants must be enforced in code and tests.
- **Follow-ups:** CONTEXT.md Asset/Clip/Project entries already align — confirm they name the
  indirection. Tests: round-trip a Project with clips intact; the split-exactness unit assert; an
  Asset-delete-under-undo test that re-inserts without ID collision.
